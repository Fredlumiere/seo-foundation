# SEO + Analytics + Audit Playbook

A reusable, end-to-end playbook for setting up SEO, analytics, automated weekly audits, non-technical fix reports, and stakeholder communication on any website.

Written for a Claude Code agent to execute alongside a human operator who handles things Claude cannot do (account creation, permission changes, credential entry, ToS approvals).

---

## End State

When this playbook is fully executed, the target site will have:

1. **Google Analytics 4** installed and collecting visitor data
2. **Google Search Console** connected, with sitemap submitted
3. **A GitHub repo** named `{project}-audit` that runs a full Lighthouse audit every Monday
4. **A dashboard** showing the latest audit (Vercel project or Cloudflare Pages — whichever fits the project's existing infra)
5. **A generated PDF** listing the top 10 fixes in priority order, with step-by-step instructions tailored to the site's CMS
6. **A stakeholder email** summarizing what was done and what needs human attention

## Scope decision (do this first)

This playbook is opinionated. It assumes a **public-facing marketing site** you want indexed in Google, tracked with GA4, and audited weekly with Lighthouse. That covers most websites, but not all. Before running Phase 1, decide with the user whether each phase applies:

### Phase 1 — GA4 + Search Console + sitemap

**Skip or substitute if any of these are true:**

- The site uses a different analytics provider (Plausible, Fathom, Mixpanel, server-side). Audit the existing setup instead — don't add GA4 on top.
- The site is intentionally noindex (internal tools, staging, private apps). Skip Search Console entirely.
- The site already has GA4 + Search Console fully configured. Audit and document the existing setup; do not duplicate tags.
- The site's sitemap is already auto-generated by the CMS or a server-side process and submitted to Search Console. Verify it's current; skip the manual generate-and-submit step.

**The default GA4 / Search Console / sitemap path documented below is the canonical setup for sites that don't have any of those in place.** If the project is using something else, document the alternative in `PROJECT-WORKSHEET.md` and adapt or skip Phase 1 accordingly.

### Phase 2 — Weekly Lighthouse audit

**Skip if:**

- The site is intentionally not optimized for performance/SEO (internal tool, dashboard, app behind auth)
- The team already has a different audit tool (DataDog Synthetics, SpeedCurve, Calibre)

Lighthouse + a static dashboard (Vercel or Cloudflare Pages) is a free, opinionated default. If the team has paid tooling, integrate with that instead.

### Phase 3 — PDF fix list

**Skip and substitute with a GitHub issue list if:**

- The audience is a technical team (developers, DevOps)
- The project has an existing issue tracker the team uses
- The fix work will be done by the team that runs CI, not by a content owner

The PDF format is for non-technical content owners (marketing, designers, agency clients). For dev teams, a markdown summary committed to the repo or a labeled set of GitHub issues is more useful.

### Phase 4 — Stakeholder email

**Skip if:**

- The user owns the comms directly and doesn't need a draft
- The team uses a different format (Slack summary, Notion doc, internal wiki)

Use the email format for external stakeholders, agency clients, or non-technical owners. For internal dev teams, a markdown summary is usually enough.

### Decision output

After this scope conversation, mark each phase in `PROJECT-WORKSHEET.md` as:

- **Run** — execute the playbook phase as documented
- **Audit existing** — verify the existing setup, document differences, don't duplicate
- **Skip** — explicitly opt out, document why

## Phase 1 — Google Analytics 4 + Search Console

**Goal:** Every page reports visits; Google knows the full URL list.

### 1.1 Check current state first

```bash
# Check if any Google tracking exists
curl -sL https://{site} | grep -Eo 'gtag|google-analytics|googletagmanager' | sort -u
# Check for Search Console verification meta
curl -sL https://{site} | grep -Eo 'google-site-verification'
```

Report findings to the user before proceeding. If GA4 or Search Console are already in place, audit them rather than duplicating.

### 1.2 Create GA4 account + property (USER ACTION)

Claude **cannot** create Google accounts or GA4 properties — account creation is a prohibited action. Instead:

- Guide the user through it via `analytics.google.com`
- Use claude-in-chrome to navigate screens and narrate ("click Create Account", "enter {site} as property name") but **do not type credentials or click submit on account creation**
- Required data to capture when done:
  - GA4 Account ID
  - GA4 Property ID
  - **Measurement ID** (format `G-XXXXXXXXXX`) — this is the one needed for the tag

Save these to memory (project type) once captured.

### 1.3 Install the gtag snippet

Write the snippet to a file the user can paste from if Claude can't insert directly:

```html
<!-- Google tag (gtag.js) -->
<script async src="https://www.googletagmanager.com/gtag/js?id=G-XXXXXXXXXX"></script>
<script>
  window.dataLayer = window.dataLayer || [];
  function gtag(){dataLayer.push(arguments);}
  gtag('js', new Date());
  gtag('config', 'G-XXXXXXXXXX');
</script>
```

Where it goes depends on the platform:

- **WordPress**: plugin like "Insert Headers and Footers", or theme `header.php`
- **Custom/static site**: directly into `<head>` in the base template
- **Next.js**: in `pages/_document.tsx` or `app/layout.tsx`
- **Webflow**: Project Settings → Custom Code → Head Code
- **Hosted page builders (Squarespace, Wix, SnapPages, etc.)**: site-wide header/footer custom-code field
- **Google Tag Manager already present**: configure GA4 tag inside GTM instead of adding the raw snippet

If the site uses Consent Mode v2 or a cookie banner, default-deny GA4 consent and update on accept. See the gtag Consent Mode v2 docs for the bootstrap pattern.

### 1.4 Verify tracking fires

Open the site in a browser, then check GA4 Realtime. Should see the visit within ~30s. If not:

```bash
# Verify the tag is actually in the rendered HTML
curl -sL https://{site} | grep 'G-XXXXXXXXXX'
```

Check GA4 Debug View for detailed events.

### 1.5 Set up Google Search Console

- Go to `search.google.com/search-console`
- Add a **URL-prefix property** for `https://{site}/` (simpler than Domain property; no DNS access needed)
- Verify via:
  - Automatic — if GA4 is already installed with the same Google account, Search Console will detect and auto-verify
  - Or HTML meta tag method: `<meta name="google-site-verification" content="..." />`
- **Link GA4 ↔ Search Console**: in GA4 → Admin → Product Links → Search Console Links

### 1.6 Find or generate a sitemap

First, check whether a sitemap already exists:

```bash
curl -sI https://{site}/sitemap.xml
curl -sI https://{site}/sitemap_index.xml
# Some CMSes use other paths
curl -sL https://{site}/robots.txt | grep -i sitemap
```

**If a sitemap exists and is current**, skip to 1.7 (submit it). Verify it lists the URLs you'd expect:

```bash
curl -sL https://{site}/sitemap.xml | grep -c '<url>'
```

**If no sitemap exists or it's stale**, generate one. The right approach depends on the platform:

- **Static / hand-built site (HTML files in a repo)**: use `scripts/generate_sitemap.py` from this repo. It walks the repo, collects indexable HTML pages, emits `sitemap.xml` with optional hreflang alternates for localized versions. Configure via CLI args or an inline config block: `BASE_URL`, `EXCLUDE_DIRS`, `EXCLUDE_FILES`, `LOCALES`. Run it before every deploy (or wire it into CI). Adapt the exclusion lists per project — every site has different "do not index" pages (404s, admin, drafts, post-signup confirmations, servlet templates with placeholder variables).

- **WordPress**: use Yoast SEO or Rank Math — both auto-generate `/sitemap_index.xml`. Don't write a custom generator.

- **Next.js**: install `next-sitemap`, configure `next-sitemap.config.js`, hook into `postbuild`. Generates `sitemap.xml` and `robots.txt` at build time.

- **Webflow**: Webflow auto-generates a sitemap if "Auto-generate Sitemap" is enabled in Project Settings → SEO. Verify it's on; otherwise upload a manual one via the same panel.

- **Hugo / Jekyll / Astro / Eleventy**: built-in sitemap generators exist for all of these. Enable in the site config rather than writing custom.

- **Server-rendered apps with dynamic URLs (lots of `/posts/{slug}` or `/products/{id}`)**: a static `sitemap.xml` won't stay current. Either generate at build time from the database/CMS, or set up a server endpoint that emits the sitemap dynamically. For very large sites (>50k URLs), shard into a sitemap index with multiple `sitemap-N.xml` files.

**Required sitemap rules** (regardless of how you generate it):

- Use absolute URLs (`https://{site}/page`, not `/page`)
- Set `<lastmod>` to a real date — Google uses it to prioritize crawl
- Exclude noindex pages explicitly. Anything in the sitemap should be indexable; anything noindex should NOT be in the sitemap.
- For multilingual sites, include `<xhtml:link rel="alternate" hreflang="..." />` entries inside each `<url>` element for every locale variant.
- Cap each sitemap file at 50,000 URLs / 50MB uncompressed; shard into a sitemap index above that.

After generation, verify:

```bash
# Sanity-check parse + URL count
curl -sL https://{site}/sitemap.xml | xmllint --noout - && echo "valid XML"
curl -sL https://{site}/sitemap.xml | grep -c '<url>'

# Confirm robots.txt points at it
curl -sL https://{site}/robots.txt | grep -i 'Sitemap:'
```

If `robots.txt` doesn't point at the sitemap, fix that — it's the canonical discovery mechanism. Add this line:

```
Sitemap: https://{site}/sitemap.xml
```

### 1.7 Submit sitemap to Search Console

- In Search Console → Sitemaps → submit the URL (e.g., `sitemap.xml` or `sitemap_index.xml`)
- Note the reported page count — you'll need this for the audit
- Submission triggers an initial crawl; follow-up crawls happen on Google's schedule (2-6 weeks for full re-index of new sites)

### 1.8 Add additional users (USER ACTION)

Claude **cannot** modify permissions in Google properties. The user must:

- GA4 → Admin → Account Access Management → Add users
- Search Console → Settings → Users and Permissions → Add user

Guide them, but they click approve.

## Phase 2 — Automated weekly audit pipeline

**Goal:** Every Monday at 10:00 UTC, a full Lighthouse audit of every page in the sitemap runs automatically and publishes to a dashboard.

### 2.0 Choose hosting for the dashboard

The dashboard is just a static site (`./dist` from unlighthouse). Two paths are both supported by this playbook:

- **Vercel** — recommended when the project being audited is already on Vercel. Reuses an existing token, no new vendor, supports Edge Middleware Basic Auth out of the box. Used by `apiant-website-audit`.
- **Cloudflare Pages** — recommended when the team already uses Cloudflare or wants a `.pages.dev` URL with no Vercel coupling. Free, public by default.

Sections **2.1–2.4** are shared. Then either follow the **Vercel path (2.5v–2.10v)** or the **Cloudflare path (2.5c–2.10c)** depending on the choice above.

#### 2.6v Optional Edge Middleware for Basic Auth — `middleware.js`

If the dashboard should not be public, drop a Basic Auth gate on every route. Skip the file entirely for a public dashboard.

```javascript
export const config = {
  matcher: ['/((?!_next/|_vercel/|favicon\\.ico).*)'],
  runtime: 'edge',
};

function timingSafeEqual(a, b) {
  if (a.length !== b.length) return false;
  let result = 0;
  for (let i = 0; i < a.length; i++) {
    result |= a.charCodeAt(i) ^ b.charCodeAt(i);
  }
  return result === 0;
}

export default function middleware(request) {
  const expectedUser = process.env.AUDIT_USER || '{project}';
  const expectedPwd = process.env.AUDIT_PASSWORD;
  if (!expectedPwd) {
    return new Response('Audit dashboard misconfigured: AUDIT_PASSWORD not set', { status: 500 });
  }
  const auth = request.headers.get('authorization');
  if (auth) {
    const [scheme, encoded] = auth.split(' ');
    if (scheme === 'Basic' && encoded) {
      try {
        const decoded = atob(encoded);
        const [user, pwd] = decoded.split(':');
        if (timingSafeEqual(user || '', expectedUser) && timingSafeEqual(pwd || '', expectedPwd)) {
          return;
        }
      } catch {}
    }
  }
  return new Response('Authentication required', {
    status: 401,
    headers: { 'WWW-Authenticate': `Basic realm="{project} audit"` },
  });
}
```

Set `AUDIT_USER` and `AUDIT_PASSWORD` as Vercel project env vars (Production scope) before the first deploy.

**Gotcha:** the "Create project if missing" step with `continue-on-error: true` is critical. On first run the Pages project doesn't exist and `wrangler pages deploy` errors with `Project not found [code 8000007]`. The create step is idempotent enough to run safely on every subsequent execution.

#### 2.6c Cloudflare API token (USER ACTION)

Claude **cannot** fill Cloudflare token creation forms — credential entry is prohibited. The user creates it via `dash.cloudflare.com` → My Profile → API Tokens → Create Token → Custom token.

Required permissions (minimum):

- **Account** → Cloudflare Pages → Edit
- **User** → User Details → Read
- **User** → Memberships → Read

Account Resources: **Include → Specific account → {user's account name}**

**Gotcha:** if Account Resources is left as "All accounts" or the scope is misconfigured, the token fails with `Authentication error [code 10000]` even though it looks valid. Verify scope by running:

```bash
curl 'https://api.cloudflare.com/client/v4/accounts' -H "Authorization: Bearer $TOKEN" | jq '.result | length'
# should return 1+, not 0
```

**Do not** create tokens with the "Edit Cloudflare Workers" template — it grants 20+ permissions (R2, KV, AI Gateway, etc.) the audit doesn't need. Always use the minimal custom scope.

## Phase 3 — Audit → non-technical PDF report

**Goal:** Turn the technical Lighthouse dashboard into a 14-page PDF a non-technical content owner can follow to actually fix the site.

For technical-audience projects, this phase can be replaced with a GitHub issue list. Skip ahead to Phase 4 in that case.

### 3.1 Download audit data

```bash
mkdir -p pdf-build
curl -s https://{project}-audit-reports.pages.dev/assets/payload.js -o pdf-build/payload.js
curl -s https://{project}-audit-reports.pages.dev/ci-result.json -o pdf-build/ci-result.json
```

`payload.js` is ~1-2 MB JavaScript with `window.__unlighthouse_payload = {reports:[...]}`. It contains full per-page Lighthouse audit data.

### 3.2 Extract prioritized findings

Parse `payload.js` with Python. Strip the `window.__unlighthouse_payload = ` prefix, then JSON-parse. For each page, iterate `report.audits` and collect audits with `score < 1`. Rank aggregated findings by `(pages_affected) × (1 - avg_score)`.

Reference extractor: `scripts/extract.py` in this repo.

```python
import json, re
raw = open('pdf-build/payload.js').read()
m = re.match(r'window\.__unlighthouse_payload\s*=\s*', raw)
data = json.loads(raw[m.end():].rstrip().rstrip(';'))
# ... aggregate audits across reports, rank, output findings.json
```

Output `findings.json` with:

- `avg_scores` (performance/accessibility/best_practices/seo)
- `worst_pages_performance` / `_accessibility` / `_seo` (top 10 each)
- `top_audits` (15 most-impactful audit types with sample affected pages)
- `pages_missing_description` / `largest_pages`

### 3.3 Screenshots of problem pages

```bash
CHROME="/Applications/Google Chrome.app/Contents/MacOS/Google Chrome"
mkdir -p pdf-build/shots
for url in "https://{site}/" "https://{site}/page1" "https://{site}/page2"; do
  slug=$(echo "$url" | sed 's|https://{site}/||; s|/|-|g')
  [ -z "$slug" ] && slug="home"
  "$CHROME" --headless --disable-gpu --hide-scrollbars \
    --screenshot="pdf-build/shots/$slug.png" \
    --window-size=1280,800 \
    --virtual-time-budget=10000 \
    "$url"
done
```

Target the 5-8 worst offenders identified in the findings JSON.

### 3.4 Build the HTML report

Reference template: `templates/report.html` in this repo.

Key design decisions (proven across multiple projects):

- **Typography**: Fraunces (serif) + Inter (sans) from Google Fonts is a strong default. For developer-audience reports, swap to Inter + JetBrains Mono.
- **Brand palette**: start with the site's primary brand colors. Two-tone (warm + accent) tends to work better than monochrome for skim-readability.
- **One fix per PDF page**, each with: priority pill, screenshot + caption, affected-page chips (clickable), "what's wrong" in plain English, "step-by-step" in a bordered callout box, optional gotcha callout
- **14 logical pages**: cover, overview + score tiles, TOC with "if you only do 3 things" summary, 10 fix pages, closing page with dashboard link + target scores

**Critical CSS for page breaks** (learned the hard way):

- Do NOT put `min-height: 11in` + `page-break-inside: avoid` on every `.page` section — it causes overflow onto half-empty next pages when content is close to the limit.
- Instead: put `page-break-inside: avoid` only on the inner `.fix` card, and `page-break-after: always` on `.page`. Drop `min-height` except on full-bleed cover and closing.
- Keep footers in-flow (`margin-top`) not `position: absolute` — absolute footers don't follow content when it overflows.
- Base font 11pt, step-list 10.5pt, tight margins (0.25em between list items).

### 3.5 Render PDF

```bash
CHROME="/Applications/Google Chrome.app/Contents/MacOS/Google Chrome"
"$CHROME" --headless --disable-gpu --no-pdf-header-footer \
  --virtual-time-budget=10000 \
  --print-to-pdf="pdf-build/{Project}-Fix-List.pdf" \
  "file://$(pwd)/pdf-build/report.html"
```

Copy to a known location for handoff:

```bash
cp pdf-build/{Project}-Fix-List.pdf ~/Desktop/
open ~/Desktop/{Project}-Fix-List.pdf
```

### 3.6 Verify page breaks

Read the PDF back with the Read tool in the harness (`pages: 1-14`). Check that:

- Every fix starts on a fresh page
- No fix has an orphan header (title on one page, body on next)
- No callout dangles alone on its own page
- Cover and closing have full-bleed background

If breaks are off, it's almost always the CSS issue in 3.4 — fix and re-render.

## Phase 4 — Stakeholder communication

**Goal:** A plain-English progress report that can be shared with whoever owns the site outcome.

For non-technical stakeholders: write an email draft to a `.txt` on Desktop (the user owns the send). For technical teams: a markdown summary committed to the project repo or a GitHub issue is more useful.

Email structure:

1. **Subject line** — one sentence, concrete
2. **What's been done** — 4 bullets: analytics, search console, access, automated audit. Each bullet includes a link to the live tool.
3. **What still needs attention** — 4-6 bullets covering the biggest technical findings in non-technical language. Link to public test tools where relevant (e.g. `pagespeed.web.dev`).
4. **A fix list** — introduce the PDF (or issue list), give a scope-dependent time estimate based on findings count, flag items that may need the CMS vendor's support team.
5. **What comes next** — preview the keyword-optimization phase so stakeholders know the foundation work is intentional.

Reference template: `templates/stakeholder-email.txt` in this repo.

- Modify user permissions / sharing / access controls

## Gotchas + lessons learned

1. **Cloudflare token scope is the #1 source of "it looks right but fails"**. Always verify with `curl .../accounts` — if it returns 0 results, the Account Resources scope is wrong. Delete and recreate with a narrow scope.

2. **Never skip `pages project create` on first deploy**. `wrangler pages deploy` does not auto-create. Use `continue-on-error: true` so the step is idempotent on reruns.

3. **`unlighthouse.config.ts` is a trap.** Use CLI flags in `package.json` scripts. Config-file imports break on CI.

4. **Commit `package-lock.json`.** Missing lockfile breaks `cache: 'npm'` in setup-node.

5. **Search Console empty-data is normal post-install.** Google re-crawls on its own schedule — 2-6 weeks. Set stakeholder expectations up front.

6. **Don't use HTML `position: absolute` footers for print PDFs.** They don't travel with content that overflows.

7. **Rotate API tokens exposed in chat.** If a token is pasted into the conversation for verification, tell the user to rotate it after.

8. **Screenshots at 1280×800 with `--virtual-time-budget=10000`** is usually enough for static sites. Iframe-heavy or SPA-heavy CMS pages may render blank — note this in the PDF caption and consider increasing the time budget.

9. **Use the free `.pages.dev` subdomain, not a custom domain**, unless the user specifically asks. Avoids DNS and certificate work.

10. **The PDF design is reusable across projects** — palette, typography, layout are token-level changes. Keep `report.html` as a template with `{project}`, `{brand-color}`, `{fix-*}` placeholders.

## Project adaptation worksheet

Before starting Phase 1, fill this out and save it alongside the project as `PROJECT-WORKSHEET.md`. Replace every `{ }` with the project's specifics.

```markdown
# {project} SEO Foundation — Adaptation Worksheet

## Site basics
- Site URL: {https://example.com}
- Project slug (used for repo names + Pages URL): {project-slug}
- CMS / platform: {wordpress | webflow | custom-static | nextjs | ...}
- Day-to-day content owner: {name + email + role}
- Audience tone for the fix report: {non-technical | technical | mixed}
- Hosting / deploy path (informational): {static, Vercel, Netlify, AWS, Cloudflare, ... }

## Existing infrastructure
- GA4 already installed: {yes/no — if yes, Measurement ID}
- Search Console already verified: {yes/no}
- Audit dashboard hosting: {Vercel | Cloudflare Pages}
- Vercel account / token: {yes/no — only needed if Vercel path}
- Cloudflare account: {yes/no — only needed if Cloudflare path}
- GitHub org/user for audit repo: {github-handle}
- Existing sitemap path: {/sitemap.xml | /sitemap_index.xml | none}
- Existing schema markup: {none | partial | full}
- Existing canonical / hreflang setup: {none | partial | full}

## Brand
- Primary brand colors (hex): {#aaaaaa, #bbbbbb}
- Typography preference: {default serif+sans | technical (Inter+JetBrains Mono) | other}
- Logo file: {path or URL}

## CMS-specific fix locations
For each Phase 3 fix type, document where to actually edit in this CMS:
- Meta description: {settings page | code template path | plugin}
- Alt text: {media library | image-block field | code template}
- H1 / page title: {page settings | code template}
- Schema markup: {plugin | hand-coded JSON-LD}
- Canonical / hreflang: {plugin | code template | server-side rewrite}

## Project-specific signals to watch for
- Does the site already have any of these that should NOT be duplicated?
  - GTM container
  - Existing GA4 / UA tags
  - Existing schema markup
  - Existing canonical / hreflang setup
- Are there any noindex pages that should stay noindex?
- Are there any servlet/template-rendered pages that need special handling (do not give them static canonicals)?
- Are there any server-side rewrites already in place (e.g., extensionless URLs, locale redirects)?

## Phase priorities for this project
For each phase, choose: **Run** | **Audit existing** | **Skip** (with reason).
- Phase 1 — GA4 + Search Console + sitemap: { Run | Audit existing | Skip — reason }
- Phase 2 — Audit pipeline + dashboard: { Run | Skip — reason }
- Phase 3 — PDF fix list: { Run | Substitute with GitHub issues | Skip — reason }
- Phase 4 — Stakeholder email or summary: { Run | Substitute with markdown summary | Skip — reason }

## Analytics / search-indexing context
- Analytics provider in use: { GA4 | Plausible | Fathom | Mixpanel | server-side | none | other }
- Search indexing posture: { public + indexed | public + intentionally noindex | private/auth-walled }
- Existing sitemap: { server-generated | static file at /sitemap.xml | none }

## Deliverables expected
- [ ] Live GA4 with realtime data
- [ ] Verified Search Console with sitemap submitted
- [ ] {project}-audit GitHub repo with passing weekly workflow
- [ ] Public dashboard at https://{project}-audit-reports.pages.dev
- [ ] {Project}-Fix-List.pdf (or GitHub issue list) on Desktop / in repo
- [ ] Stakeholder email draft on Desktop (or markdown summary in repo)
```

Start every new project by running the Phase 1.1 current-state check and filling in the worksheet. Then walk Phase 1 → 4 in order, skipping phases the user opts out of.
